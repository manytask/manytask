# Universal MR Review Bot — Product Specification

**Тикет:** EDUCATION-57552  
**Дата:** 2026-04-12  
**Статус:** Утверждена, готова к планированию реализации

---

## 1. Цель и контекст

**Что строим:** Universal MR Review Bot — сервис, позволяющий любому курсу на платформе Manytask организовать проверку домашних заданий через GitLab Merge Requests. Преподаватель ревьюит код студента и ставит оценку командой в комментарий — бот сохраняет её в Manytask.

**Почему сейчас:** Python YSDA уже использует `pythonbot-2025`, но он жёстко привязан к одному курсу. Расширение нужно для курсов Go и C++; стратегическая цель — перевод курса Алгоритмов с Gerrit на GitLab + Manytask.

**Ключевые ограничения:**
- Бот — честный микросервис: без прямого доступа к БД Manytask, только через API
- Архитектура абстрагирована от git-хостинга (GitLab сегодня, SourceCraft параллельно и в будущем)
- Конфигурация через `manytask.yml` (концепция "course is a code"); UI не нужен — преподаватели технически грамотны

**Критерий успеха MVP:** хотя бы один курс кроме Python (Go или C++) использует бота в продакшне.  
**Стратегический успех:** курс Алгоритмов переведён с Gerrit на GitLab + Manytask через этого бота.

## 2. Роли

| Роль | Описание |
|------|----------|
| **Студент** | Отправляет решение через MR, получает автоматическую и ручную обратную связь |
| **Преподаватель** | Ревьюит MR, выставляет оценку командой в комментарий; может редактировать конфиг курса и деплоить его |
| **Платформенная команда** | Поддерживает инфраструктуру бота; один инстанс для всех курсов |

## 3. Пользовательские сценарии

### Сценарий 1: Студент сдаёт задачу

1. Студент создаёт MR в GitLab с решением (в папке, заданной конфигом курса)
2. Бот при очередном polling (≤15 мин) обнаруживает новый MR
3. Бот запускает **pre-review checklist** — произвольный код из конфига курса (тесты прошли, файлы в правильном месте, нет запрещённых расширений и т.д.)
4. Бот оставляет в MR комментарий с результатами checklist (passed/failed по каждому пункту)
5. В Manytask у задачи появляется статус **"На проверке"** и прямая ссылка на MR

### Сценарий 2: Преподаватель ревьюит и выставляет оценку

1. Преподаватель находит MR через GitLab (фильтр по лейблу курса, хронологическая сортировка)
2. Смотрит код, оставляет комментарии
3. Пишет финальный комментарий с оценкой в формате из конфига курса (например, `Score: 85`)
4. Бот при следующем polling видит комментарий, верифицирует, что автор — преподаватель (GitLab Developer+ access level в MVP)
5. Бот отправляет оценку в Manytask через `POST /api/<course>/report`
6. Статус задачи в Manytask меняется на **"Проверено"**

### Сценарий 3: Преподаватель меняет оценку

1. Преподаватель оставляет новый комментарий с оценкой в том же формате
2. Бот обнаруживает новый комментарий и перезаписывает оценку в Manytask (override)

### Сценарий 4: Подключение / обновление курса

1. Любой преподаватель курса редактирует `manytask.yml` — добавляет или обновляет секцию ручной проверки (задачи, папки, checklist-скрипты, формат оценки)
2. При деплое курса CI/CD автоматически вызывает API-ручку бота (`POST /courses`) с актуальным конфигом
3. Бот обновляет конфиг курса в Redis и (если курс новый) добавляет его в цикл polling
4. Отдельный инстанс бота на курс **не поднимается** — один инстанс обслуживает все курсы

## 4. Архитектура

```
manytask.yml (курс)
       │ deploy (CI/CD)
       ▼
  Bot HTTP API
  POST /courses  ──────────────────────► Redis
                                          │
                                    конфиги курсов
                                    обработанные MR
                                    (idempotency set)

  Polling Worker (каждые 15 мин)
       │
       ├── Git Hosting Adapter (GitLab / SourceCraft)
       │      читает MR, пишет комментарии и лейблы
       │
       ├── Checklist Runner
       │      исполняет произвольный код из конфига курса
       │
       └── Manytask API Client
              ├── GET  /api/<course>/deadlines       (новый)
              ├── GET  /api/<course>/is_admin        (новый, v2)
              └── POST /api/<course>/report          (существующий)
```

**Ключевые архитектурные решения:**

| Вопрос | Решение | Обоснование |
|--------|---------|-------------|
| Деплой | Один Docker-контейнер + Redis | Нагрузка IO-bound, хватит небольшой VM (4–8 CPU) |
| Триггер | Polling каждые 15 мин | Webhooks требуют публичный URL; polling справляется с пиком ~4000 MR |
| Хранение конфигов | Redis | Уже используется для idempotency; низкая нагрузка |
| Idempotency | Redis set обработанных comment ID | Корректная работа после рестарта; хранятся ID комментариев, а не MR — чтобы новый score-комментарий (override) обрабатывался корректно |
| Абстракция хостинга | Интерфейс над GitLab/SourceCraft | Курсы могут одновременно жить на разных платформах |
| Права преподавателя | MVP: GitLab Developer+ access level | Достаточно для MVP; v2 — `is_course_admin` через Manytask API |
| Безопасность checklist | Eval + liveness-проба с автоперезапуском | Бот для внутреннего использования; преподавателям доверяем |
| Username matching | GitLab username = Manytask username | Гарантировано историческим контрактом, сохраняется при переезде на SourceCraft |

## 5. Конфигурация курса (пример `manytask.yml`)

```yaml
# Секция ручной проверки внутри manytask.yml

mr_review:
  gitlab_group: python/students-2025-fall
  score_comment_pattern: "Score: {score}"  # формат комментария преподавателя

  tasks:
    compgraph:
      folder: "09.2.HW2/tasks/compgraph"
      manual_review: true
      checklist: |
        # Произвольный Python-код (язык checklist — Python)
        # Доступны: mr, gitlab_client, manytask_client
        assert pipeline_passed(mr), "CI pipeline must pass"
        assert no_forbidden_files(mr, ['.csv', '.db']), "Forbidden file types found"
```

## 6. Интеграция с Manytask

### Статусы задачи в Manytask UI

При наличии флага `manual_review: true` у задачи в Manytask отображается:
- Статус: `На проверке` / `Проверено` / `Ждёт исправлений`
- Ссылка на MR студента

### Новые API endpoints (требуют разработки в Manytask)

| Endpoint | Метод | Описание |
|----------|-------|----------|
| `/api/<course>/deadlines` | GET | Дедлайны задач курса в машиночитаемом формате |
| `/api/<course>/is_admin?username=X` | GET | Проверка, является ли пользователь преподавателем курса (v2) |

### Существующие endpoints (используются без изменений)

| Endpoint | Метод | Описание |
|----------|-------|----------|
| `/api/<course>/report` | POST | Сохранение оценки студента |

## 7. MVP Scope

### В MVP

| Фича | Детали |
|------|--------|
| Polling GitLab | Каждые 15 мин, один инстанс для всех курсов |
| Pre-review checklist | Eval произвольного Python-кода из конфига курса |
| Выставление оценки | Преподаватель пишет `Score: N` → бот сохраняет в Manytask |
| Override оценки | Новый комментарий с оценкой перезаписывает предыдущую |
| Статусы в Manytask | "На проверке" / "Проверено" / "Ждёт исправлений" + ссылка на MR |
| Конфигурация курса | Через `manytask.yml`, загружается в Redis при деплое (`POST /courses`) |
| Права преподавателя | GitLab Developer+ access level |
| Idempotency | Redis хранит set обработанных comment ID |
| Абстракция хостинга | Интерфейс над GitLab / SourceCraft |
| Новые API Manytask | `GET /deadlines` endpoint |
| Liveness-проба | Автоперезапуск при падении checklist |

### Беклог (вне MVP)

| Фича | Причина отложить |
|------|-----------------|
| Webhook-триггер | Требует публичный URL + DNS записи |
| Push-уведомления преподавателям (Telegram/Slack) | Достаточно GitLab-фильтров по лейблу |
| UI дашборд для преподавателей | Достаточно GitLab |
| Автоматические штрафы за дедлайны | Логику штрафов решает преподаватель |
| Проверка прав через Manytask `is_course_admin` | GitLab access level достаточен в MVP |
| Поддержка множественных дедлайнов внутри задачи | Отложено |

---

## 8. Риски

| Риск | Вероятность | Влияние | Митигация |
|------|-------------|---------|-----------|
| Форк-бомба в checklist-коде | Низкая | Высокое | Liveness-проба с автоперезапуском |
| Рассинхронизация дедлайнов (Manytask vs бот) | Устранён | — | Источник истины — только Manytask API |
| Бот обработает MR повторно при рестарте | Устранён | — | Redis idempotency set |
| Курс переедет на SourceCraft, бот сломается | Устранён | — | Абстракция хостинга в архитектуре |

---

## 9. Зависимости

- **Manytask:** добавить `GET /api/<course>/deadlines`; добавить статусы MR в UI задачи
- **CI/CD курсов:** при деплое `manytask.yml` вызывать `POST /courses` на боте
- **Git-хостинг:** доступ к GitLab API с правами чтения MR и записи комментариев
